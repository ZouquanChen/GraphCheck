# GraphCheck 数据处理学习笔记

> 这份笔记先只聚焦一个主题：`数据是怎样一步步从原始样本变成训练时可用的图 batch`。  
> 建议你一边读文档，一边对照源码里的 `graph_build.py`、`dataset/utils/dataset.py`、`dataset/utils/collate.py`、`train.py`。

## 1. 先建立整体印象：数据处理在项目里的位置

如果把 GraphCheck 整个项目看成一条流水线，那么数据处理部分负责做这几件事：

1. 从 `dataset/*.pkl` 里读出原始样本；
2. 把 `doc_kg` 和 `claim_kg` 里的三元组整理成图结构；
3. 把节点文本和边文本编码成向量；
4. 保存成 PyTorch Geometric 可直接读取的 `.pt` 图对象；
5. 生成 `train / val / test` 划分索引；
6. 在训练阶段重新把这些图和原始文本拼回来，交给 `DataLoader` 和模型。

你可以先把它记成下面这条主线：

```text
dataset/<name>/<name>.pkl
    -> graph_build.py
    -> extracted_KG/<name>/nodes + edges + graphs + split
    -> KGDataset
    -> collate_fn
    -> train.py / inference.py
```

这一条主线里，最重要的不是记住每个函数名，而是记住每一步的“产物”是什么。

## 2. 原始数据长什么样

项目 README 里已经说明，每条样本至少包含这几个字段：

- `doc_text`：原始文档文本
- `claim_text`：待核查的声明
- `doc_kg`：从文档抽取出的知识图谱三元组
- `claim_kg`：从声明抽取出的知识图谱三元组
- `label`：标签，`1` 表示 `support`，`0` 表示 `unsupport`

- 一条样本本质上是“文档 + 声明 + 两张待构图的 KG + 一个标签”；
- 后面的预处理并不会丢掉这些信息，而是把其中的 KG 部分变成图对象；

这里顺便提醒你一个阅读源码时很容易混淆的点：

- 仓库里当前能直接看到的原始 `.pkl` 示例，位于 `dataset/<dataset>/<dataset>.pkl`；
- 但 `graph_build.py` 和 `dataset/utils/dataset.py` 的路径拼接，主要围绕 `dataset/extracted_KG/<dataset>/...` 展开；
- 所以你学习时最好把它们区分成两类目录：
  - 原始数据目录；
  - 预处理输出目录。

换句话说，源码主线关注的是“预处理之后的数据如何被训练代码读取”，而不只是“原始 `.pkl` 放在哪里”。

## 3. `graph_build.py` 在做什么

`graph_build.py` 是整个数据处理阶段的核心入口。它主要做三件事：

1. 把 KG 三元组改写成节点表和边表；
2. 把节点/边文本编码成向量并保存为图对象；
3. 生成训练、验证、测试集划分。

从代码实现看，这个脚本最后会把结果组织到 `dataset/extracted_KG/<dataset>/...` 这棵目录下，因此你后面看到的 `nodes/`、`edges/`、`graphs/`、`split/` 都是在这个根目录下面。

### 3.1 `textualize_graph()`：把三元组变成节点表和边表

先看关键代码：

```python
def textualize_graph(graph):
    if not graph:
        nodes = pd.DataFrame(columns=['node_attr', 'node_id'])
        edges = pd.DataFrame(columns=['src', 'edge_attr', 'dst'])
        return nodes, edges

    nodes = {}
    edges = []

    for tri in graph:
        src, edge_attr, dst = tri

        src = src.lower().strip() if src else " "
        edge_attr = edge_attr.lower().strip() if edge_attr else " "
        dst = dst.lower().strip() if dst else " "

        if src not in nodes:
            nodes[src] = len(nodes)
        if dst not in nodes:
            nodes[dst] = len(nodes)

        edges.append({
            'src': nodes[src],
            'edge_attr': edge_attr,
            'dst': nodes[dst],
        })

    nodes = pd.DataFrame(nodes.items(), columns=['node_attr', 'node_id'])
    edges = pd.DataFrame(edges)
    return nodes, edges
```

这段代码可以理解成“把文本形式的 KG 三元组，标准化成图结构的中间表示”：

- 输入里的一个三元组是 `(head, relation, tail)`；
- `src` 和 `dst` 会先转成小写、去掉首尾空格；
- `nodes` 字典负责给每个实体分配一个唯一的整数 `node_id`；
- `edges` 里保存边的信息，即“从哪个节点出发、边的文字是什么、指向哪个节点”；
- 函数最后返回两个表：
  - `nodes`：每个节点的编号和文本属性；
  - `edges`：每条边的起点、终点和关系文本。

这里你要特别注意一点：

- 项目并不是直接把字符串三元组喂给 GNN；
- 它先把字符串整理成“节点表 + 边表”；
- 后面再单独把这些文本编码成向量。

### 3.2 `step_one()`：为每条样本保存节点 CSV 和边 CSV

关键代码如下：

```python
def step_one():
    os.makedirs(f'{path}/{data_name}/nodes/claim', exist_ok=True)
    os.makedirs(f'{path}/{data_name}/nodes/doc', exist_ok=True)
    os.makedirs(f'{path}/{data_name}/edges/claim', exist_ok=True)
    os.makedirs(f'{path}/{data_name}/edges/doc', exist_ok=True)

    for i, (claim_kg, doc_kg) in enumerate(tqdm(zip(claim_kgs, doc_kgs), total=len(dataset))):
        claim_nodes, claim_edges = textualize_graph(claim_kg)
        claim_nodes.to_csv(f'{path}/{data_name}/nodes/claim/{i}.csv', index=False, columns=['node_id', 'node_attr'])
        claim_edges.to_csv(f'{path}/{data_name}/edges/claim/{i}.csv', index=False, columns=['src', 'edge_attr', 'dst'])

        doc_nodes, doc_edges = textualize_graph(doc_kg)
        doc_nodes.to_csv(f'{path}/{data_name}/nodes/doc/{i}.csv', index=False, columns=['node_id', 'node_attr'])
        doc_edges.to_csv(f'{path}/{data_name}/edges/doc/{i}.csv', index=False, columns=['src', 'edge_attr', 'dst'])
```

你可以把 `step_one()` 理解成“把每个样本拆成可落盘的结构化文件”：

- 对第 `i` 条样本，项目会分别处理 `claim_kg` 和 `doc_kg`；
- 每个图都会得到两类文件：
  - `nodes/.../{i}.csv`
  - `edges/.../{i}.csv`
- 这样做的好处是，后面的图编码步骤可以直接按文件逐个读取，不需要重复解析原始三元组。

这一阶段的产物大致像这样：

```text
dataset/extracted_KG/MiniCheck_Train/
  nodes/claim/0.csv
  nodes/doc/0.csv
  edges/claim/0.csv
  edges/doc/0.csv
```

### 3.3 `step_two()`：把节点/边文本编码成向量图

关键代码如下：

```python
def step_two():
    def _encode_graph():
        os.makedirs(f'{path}/{data_name}/graphs/claim', exist_ok=True)
        for i in tqdm(range(len(dataset))):
            nodes = pd.read_csv(f'{path}/{data_name}/nodes/claim/{i}.csv')
            edges = pd.read_csv(f'{path}/{data_name}/edges/claim/{i}.csv')
            x = text2embedding(model, tokenizer, device, nodes.node_attr.tolist())
            e = text2embedding(model, tokenizer, device, edges.edge_attr.tolist())
            edge_index = torch.LongTensor([edges.src, edges.dst])
            data = Data(x=x, edge_index=edge_index, edge_attr=e, num_nodes=len(nodes))
            torch.save(data, f'{path}/{data_name}/graphs/claim/{i}.pt')
```

这一步最关键，因为它第一次把“文本图”变成“向量图”：

- `nodes.node_attr.tolist()` 是所有节点的文本，比如实体名；
- `edges.edge_attr.tolist()` 是所有边的文本，比如关系名；
- `text2embedding(...)` 会把这些文本编码成向量；
- `edge_index` 是 PyG 常见的图结构格式，记录边的连接关系；
- 最终用 `Data(...)` 组装成 PyTorch Geometric 的图对象；
- 然后保存为 `.pt` 文件。

这里生成的 `Data` 对象通常包含：

- `x`：节点特征矩阵，形状大致是 `[num_nodes, 1024]`
- `edge_index`：边连接关系，形状是 `[2, num_edges]`
- `edge_attr`：边特征矩阵，形状大致是 `[num_edges, 1024]`
- `num_nodes`：节点数

文档图也会走同样流程，只是保存在 `graphs/doc/` 下面。

所以这一步结束后，项目真正可供训练读取的核心文件是：

```text
dataset/extracted_KG/<dataset>/graphs/claim/<i>.pt
dataset/extracted_KG/<dataset>/graphs/doc/<i>.pt
```

你后面读 `KGDataset` 时会发现，它就是从这些 `.pt` 文件里把图读回来。

## 4. 训练集、验证集、测试集是怎么划分的

这一部分在 `generate_split()` 里完成，关键代码如下：

```python
def generate_split(num_nodes, path):
    indices = np.arange(num_nodes)
    train_indices, temp_data = train_test_split(indices, test_size=0.4, random_state=42)
    val_indices, test_indices = train_test_split(temp_data, test_size=0.5, random_state=42)
```

这三行代码要这样理解：

- 先生成从 `0` 到 `num_nodes - 1` 的样本索引；
- 第一次切分时，拿出 `40%` 作为临时集合 `temp_data`，剩下 `60%` 作为训练集；
- 第二次再把这 `40%` 平分成验证集和测试集；
- 所以最终比例是：
  - `train = 60%`
  - `val = 20%`
  - `test = 20%`

继续看保存逻辑：

```python
with open(f'{path}/train_indices.txt', 'w') as file:
    file.write('\n'.join(map(str, train_indices)))

with open(f'{path}/val_indices.txt', 'w') as file:
    file.write('\n'.join(map(str, val_indices)))

with open(f'{path}/test_indices.txt', 'w') as file:
    file.write('\n'.join(map(str, test_indices)))
```

说明项目不是每次训练时都临时随机切分，而是：

- 预处理阶段生成一次；
- 结果写入 `split/*.txt`；
- 训练时直接读取这些索引文件复用。

这些 split 文件通常位于：

```text
dataset/extracted_KG/<dataset>/split/train_indices.txt
dataset/extracted_KG/<dataset>/split/val_indices.txt
dataset/extracted_KG/<dataset>/split/test_indices.txt
```

这里还有两个学习时容易忽略的点：

- `random_state=42` 让切分可复现；
- 代码里没有 `stratify=labels`，所以这不是严格的分层采样，而是普通随机划分。

---
*_indices.txt 里保存的内容，本质上就是“样本编号列表”。
- 例如 train_indices.txt 里每一行都是一个整数，表示“这个样本属于训练集”
- 这些整数来自 np.arange(num_nodes)，也就是从 0 到 样本总数-1 的索引，见 graph_build.py:196
- 然后代码用 train_test_split(...) 把这些索引切成三份，分别写入：
  - train_indices.txt
  - val_indices.txt
  - test_indices.txt
  见 graph_build.py:198、graph_build.py:200、graph_build.py:215
你可以把它理解成这样：
假设数据集一共 5 条样本
原始索引就是：0, 1, 2, 3, 4
切分后可能变成：
train_indices.txt -> 0
                     2
                     4
val_indices.txt   -> 1
test_indices.txt  -> 3
为什么只保存“索引”，不直接保存整条数据？
- 因为整条数据已经在别处了：
  - 原始文本和标签在 .pkl 里
  - 图对象在 graphs/claim/<index>.pt 和 graphs/doc/<index>.pt
- 这些文件都共享同一个样本编号体系，所以只要记住“哪些编号属于 train/val/test”就够了
- 这样更省空间，也方便复现同一套切分
这些文件后面是怎么用的？
- KGDataset.get_idx_split() 会把这三个文件重新读回来，见 dataset/utils/dataset.py:53
- train.py 再按这些索引挑出训练集、验证集、测试集，见 train.py:33、train.py:36
所以一句话概括：
- *_indices.txt 的内容就是“哪些样本编号被分到哪个集合里”，作用是固定数据划分，方便训练时按编号取样本

## 5. `KGDataset` 怎样把文件重新组装成一条训练样本

真正训练时使用的数据集类在 `dataset/utils/dataset.py` 里。

### 5.1 初始化：先把原始元数据读进来

```python
def get_dataset(dataset_name):
    result_df = pd.read_pickle(f'{PATH}/extracted_KG/{dataset_name}/{dataset_name}.pkl')
    docs = result_df['doc_text'].values
    claims = result_df['claim_text'].values
    doc_kgs = result_df['doc_kg'].values
    claim_kgs = result_df['claim_kg'].values
    labels = result_df['label'].values
    return docs, claims, doc_kgs, claim_kgs, labels
```

这里说明 `KGDataset` 并不只读图文件，它还保留了原始文本和标签：

- `docs` 和 `claims` 后面会被拼成 prompt；
- `doc_kgs` 和 `claim_kgs` 的文本形式也会保留，方便后续使用；
- `labels` 决定训练目标是 `support` 还是 `unsupport`。

### 5.2 `__getitem__()`：按索引取出一条完整样本

```python
def __getitem__(self, index):
    doc, claim, doc_kgs_text, claim_kgs_text, label = (
        self.docs[index], self.claims[index], self.doc_kgs[index], self.claim_kgs[index], self.labels[index]
    )
    text = f'{self.prompt}\nClaim: {claim}\nDocument: {doc}'
    claim_kg = torch.load(f'{PATH}/extracted_KG/{self.dataset_name}/graphs/claim/{index}.pt')
    doc_kg = torch.load(f'{PATH}/extracted_KG/{self.dataset_name}/graphs/doc/{index}.pt')
    if label == 1:
        label = 'support'
    else:
        label = 'unsupport'
```

这几行代码非常关键，因为它说明一条训练样本实际上由三部分组成：

1. 文本输入：`Claim + Document` 拼成的 prompt；
2. 图输入：`claim_kg.pt` 和 `doc_kg.pt` 两个图对象；
3. 标签：最终被转成自然语言形式的 `support / unsupport`。

也就是说，GraphCheck 不是一个“只吃图”的模型，也不是一个“只吃文本”的模型，而是文本和图同时输入。

再看它返回的字典：

```python
return {
    'id': index,
    'label': label,
    'claim_kg': claim_kg,
    'doc_kg': doc_kg,
    'claim_kg_text': claim_kgs_text,
    'doc_kg_text': doc_kgs_text,
    'text': text,
    'index': index,
    'dataset': self.dataset_name
}
```

这相当于告诉你：模型后面拿到的不是单独的一个 tensor，而是一份“多模态样本包”，里面同时有：

- 文本；
- 两张图；
- 标签；
- 若干辅助字段。

### 5.3 `get_idx_split()`：读取预处理阶段保存的划分文件

```python
def get_idx_split(self):
    with open(f'{PATH}/extracted_KG/{self.dataset_name}/split/train_indices.txt', 'r') as file:
        train_indices = [int(line.strip()) for line in file]
    with open(f'{PATH}/extracted_KG/{self.dataset_name}/split/val_indices.txt', 'r') as file:
        val_indices = [int(line.strip()) for line in file]
    with open(f'{PATH}/extracted_KG/{self.dataset_name}/split/test_indices.txt', 'r') as file:
        test_indices = [int(line.strip()) for line in file]
    return {'train': train_indices, 'val': val_indices, 'test': test_indices}
```

这个函数的作用很直接：

- 训练时不再重新随机切分；
- 直接读预处理阶段已经保存好的索引；
- 返回一个字典，供 `train.py` 构造训练、验证、测试子集。

## 6. `collate_fn` 怎样把多条样本合并成一个 batch

如果 `Dataset.__getitem__()` 返回的是一个包含图对象和文本的字典，那么 `DataLoader` 默认的拼接方式往往不够用，因此项目定义了自己的 `collate_fn`：

```python
from torch_geometric.data import Batch

def collate_fn(original_batch):
    batch = {}
    for k in original_batch[0].keys():
        batch[k] = [d[k] for d in original_batch]
    if 'claim_kg' in batch:
        batch['claim_kg'] = Batch.from_data_list(batch['claim_kg'])
    if 'doc_kg' in batch:
        batch['doc_kg'] = Batch.from_data_list(batch['doc_kg'])
    return batch
```

这段代码可以拆成两层理解：

- 第一层：先把每个字段都收集成列表；
  - 比如所有 `text` 放成一个列表；
  - 所有 `label` 放成一个列表；
  - 所有 `claim_kg` 放成一个图对象列表。
- 第二层：对图字段做特殊处理；
  - `Batch.from_data_list(...)` 会把多张图合并成一个 PyG batch；
  - 这样后面的 GNN 才能一次处理一个 mini-batch 的图。

所以 `collate_fn` 的意义是：

- 文本字段保持 Python 列表形式；
- 图字段变成 PyG 的批量图结构；
- 最后统一打包成一个 `batch` 字典交给模型。

## 7. 训练入口 `train.py` 怎样使用这些数据

到了 `train.py`，数据处理阶段生成的产物才真正被串起来。

关键代码如下：

```python
dataset = KGDataset(args.train_dataset)
idx_split = dataset.get_idx_split()

train_dataset = [dataset[i] for i in idx_split["train"]]
val_dataset = [dataset[i] for i in idx_split["val"]]
test_dataset = [dataset[i] for i in idx_split["test"]]

train_loader = DataLoader(
    train_dataset,
    batch_size=args.batch_size,
    shuffle=True,
    collate_fn=collate_fn,
)
```

这段逻辑建议你按顺序理解：

1. 先创建一个完整的 `KGDataset`；
2. 从 `split/*.txt` 读出 train/val/test 的索引；
3. 按这些索引挑出三个子集；
4. 再交给 `DataLoader`；
5. `DataLoader` 在每个 batch 上调用 `collate_fn`；
6. 最终 batch 里同时有文本和两张图。

也就是说，训练阶段并不会再去做 KG 构建和文本编码，那些工作已经在 `graph_build.py` 里提前完成了。

你还可以顺便注意几个训练细节：

- `train_loader` 使用 `shuffle=True`，训练时会打乱顺序；
- `val_loader` 和 `test_loader` 都是 `shuffle=False`；
- 说明项目只在训练阶段打乱样本，验证和测试保持稳定顺序。

- `x` 是节点向量；

- `edge_attr` 是边向量；
- 这已经是 GNN 可以直接消费的数据格式。

## 9. 这一阶段学完后，你脑中应该留下什么

如果你已经读到这里，建议你最后检查自己有没有建立下面这条因果链：

- 原始 `.pkl` 里既有文本，也有 KG 三元组；
- `graph_build.py` 把 KG 三元组转成节点/边表，再编码成图对象；
- 同时生成 `train / val / test` 划分索引；
- `KGDataset` 在训练时把文本、图、标签重新拼成一条样本；
- `collate_fn` 把多条样本打成 batch；
- `train.py` 只负责读取这些产物，不负责重新做预处理。

如果这条链你已经能复述出来，说明 GraphCheck 的数据处理主线你已经掌握得差不多了。
