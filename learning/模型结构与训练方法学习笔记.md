# GraphCheck 模型结构与训练方法学习笔记

> 这份笔记接在 `learning/数据处理学习笔记.md` 后面。  
> 上一份笔记回答的是：`数据怎样变成图 batch`；这一份要回答的是：`这些图最后怎样进入模型，以及训练时到底是谁在学习`。

建议你一边看这份文档，一边对照 `model/graphcheck.py`、`model/gnn.py`、`train.py`、`src/utils.py`、`src/ckpt.py`。

## 1. 先建立整体印象：GraphCheck 不是纯 LLM，也不是纯 GNN

如果只看名字，很多人会以为 GraphCheck 是“拿一个大模型，然后额外挂个图网络”。

这个理解不算错，但还不够具体。更准确地说，当前实现里的 GraphCheck 可以拆成三层：

1. 一个冻结的因果语言模型（LLM）；
2. 一个负责编码 `claim_kg` 和 `doc_kg` 的 GNN；
3. 一个把图表示映射到 LLM 词向量空间的 `projector`。

你可以先把它记成这样：

```text
claim_kg/doc_kg 图
    -> GNN 编码
    -> 图级别向量
    -> projector 映射到 LLM embedding 空间
    -> 作为额外 embedding 拼到文本前面
    -> 冻结 LLM 生成 support / unsupport
```

所以它不是“GNN 先分类，LLM 再润色”，也不是“LLM 单独读文本，图只是辅助特征”。

它真正做的是：

- 先把两张图压成两个向量；
- 再把这两个向量当成额外的输入 embedding；
- 最后交给冻结的 LLM 继续做生成。

这意味着 GraphCheck 的训练目标，本质上是让图编码器学会产出一种“对 LLM 有帮助的图提示”。

## 2. 模型总结构：`GraphCheck` 里到底有哪些部件

先看 `model/graphcheck.py` 里的核心初始化逻辑：

```python
self.tokenizer = AutoTokenizer.from_pretrained(args.llm_model_path, use_fast=False)

model = AutoModelForCausalLM.from_pretrained(
    args.llm_model_path,
    torch_dtype=torch.float16,
    low_cpu_mem_usage=True,
    **kwargs
)

for name, param in model.named_parameters():
    param.requires_grad = False

self.model = model
self.word_embedding = self.model.model.get_input_embeddings()

self.graph_encoder = load_gnn_model[args.gnn_model_name](
    in_channels=args.gnn_in_dim,
    out_channels=args.gnn_hidden_dim,
    hidden_channels=args.gnn_hidden_dim,
    num_layers=args.gnn_num_layers,
    dropout=args.gnn_dropout,
    num_heads=args.gnn_num_heads,
).to(self.model.device)

self.projector = nn.Sequential(
    nn.Linear(args.gnn_hidden_dim, 2048),
    nn.Sigmoid(),
    nn.Linear(2048, self.word_embedding.weight.shape[1]),
).to(self.model.device)
```

这段代码里最重要的是 4 件事：

- `self.model` 是真正的 HuggingFace 因果语言模型；
- 这个 LLM 的所有参数都被 `requires_grad = False` 冻结了；
- `self.graph_encoder` 负责把图里的节点和边编码成图表示；
- `self.projector` 负责把图表示变成和词向量同维度的向量。

这里你要特别注意“冻结 LLM”这件事，因为它直接决定了训练方法。

项目不是在微调整个大模型，而是在做一种更轻量的训练：

- LLM 提供已有的语言理解和生成能力；
- GNN 负责把 KG 信息抽成图表示；
- projector 负责把图表示翻译成 LLM 能读懂的 embedding。

换句话说，当前实现里真正会学习的，不是大模型主体，而是“图到语言 embedding 的桥”。

## 3. `model/gnn.py` 在做什么：先编码节点，再汇总成图向量

GraphCheck 支持两种图编码器：`GAT` 和 `GraphTransformer`。

对应代码在 `model/gnn.py`：

```python
class GAT(torch.nn.Module):
    def __init__(self, in_channels, hidden_channels, out_channels, num_layers, dropout, num_heads=4):
        ...
        self.convs.append(GATConv(in_channels, hidden_channels, heads=num_heads, concat=False))
        ...
        self.convs.append(GATConv(hidden_channels, out_channels, heads=num_heads, concat=False))

    def forward(self, x, edge_index, edge_attr):
        for i, conv in enumerate(self.convs[:-1]):
            x, attn_weights = conv(x, edge_index=edge_index, edge_attr=edge_attr, return_attention_weights=True)
            x = self.bns[i](x)
            x = F.relu(x)
            x = F.dropout(x, p=self.dropout, training=self.training)

        x, attn_weights = self.convs[-1](x, edge_index=edge_index, edge_attr=edge_attr, return_attention_weights=True)
        return x, edge_attr
```

这说明图编码器吃进去的是：

- `x`：节点特征；
- `edge_index`：边连接关系；
- `edge_attr`：边特征。

而这些输入，正是上一份数据处理笔记里 `graph_build.py` 预先保存到 `.pt` 图对象里的内容。

所以从输入来源上说：

- 节点特征不是随机初始化的 ID embedding；
- 边特征也不是离散编号；
- 它们已经是文本编码后的 1024 维向量。

也就是说，GNN 处理的不是“结构空图”，而是“带语义向量的图”。

## 4. `encode_graphs()`：两张图最后怎样变成两个图级别向量

`GraphCheck` 并不会把整张图的所有节点 embedding 都一股脑塞进 LLM。当前实现更克制：它先把每张图压成一个图级别向量。

核心代码如下：

```python
def encode_graphs(self, data):
    claim_kg = data['claim_kg'].to(self.model.device)
    doc_kg = data['doc_kg'].to(self.model.device)

    claim_n_embeds, _ = self.graph_encoder(claim_kg.x, claim_kg.edge_index.long(), claim_kg.edge_attr)
    doc_n_embeds, _ = self.graph_encoder(doc_kg.x, doc_kg.edge_index.long(), doc_kg.edge_attr)

    if claim_kg.batch is not None:
        claim_embeds = scatter(claim_n_embeds, claim_kg.batch, dim=0, reduce='mean')
    else:
        claim_embeds = claim_n_embeds.mean(dim=0, keepdim=True)

    if doc_kg.batch is not None:
        doc_embeds = scatter(doc_n_embeds, doc_kg.batch, dim=0, reduce='mean')
    else:
        doc_embeds = doc_n_embeds.mean(dim=0, keepdim=True)

    return claim_embeds, doc_embeds
```

这段代码建议你拆成两步理解。

### 4.1 第一步：得到节点级别表示

`self.graph_encoder(...)` 的输出 `claim_n_embeds` 和 `doc_n_embeds`，本质上还是“每个节点一个向量”。

也就是说，经过 GNN 后，模型首先得到的是：

- `claim` 图里每个节点的新表示；
- `doc` 图里每个节点的新表示。

### 4.2 第二步：做图级别池化

后面的 `scatter(..., reduce='mean')` 才是关键。

它的作用是：

- 按 batch 里的图编号，把属于同一张图的节点向量聚合起来；
- 聚合方式是平均池化；
- 最后每张图只保留一个向量。

所以到了 `encode_graphs()` 的返回值阶段：

- 每条样本的 `claim_kg` 会被压成一个向量；
- 每条样本的 `doc_kg` 也会被压成一个向量。

这说明当前实现的图注入粒度非常明确：

- 不是“每个节点都变成一个 token”；
- 也不是“每条边都单独送进 LLM”；
- 而是“一张 claim 图一个向量，一张 doc 图一个向量”。

## 5. 图是怎么注入 LLM 的：不是拼字符串，而是直接拼 embedding

这一部分是整个模型里最值得你盯住的地方。

很多人第一次看这种模型，会以为图信息是被转成一段文本 prompt，然后让 LLM 自己读。当前实现不是这样。

先看 `forward()` 里的核心代码：

```python
texts = self.tokenizer(data["text"], add_special_tokens=False)
labels = self.tokenizer(data["label"], add_special_tokens=False)

eos_tokens = self.tokenizer(EOS, add_special_tokens=False)
eos_user_tokens = self.tokenizer(EOS_USER, add_special_tokens=False)
bos_embeds = self.word_embedding(self.tokenizer(BOS, add_special_tokens=False, return_tensors='pt').input_ids[0].cuda())

claim_embeds, doc_embeds = self.encode_graphs(data)
claim_embeds = self.projector(claim_embeds)
doc_embeds = self.projector(doc_embeds)
```

先发生的事是：

- 文本 prompt 被 tokenizer 编码；
- 标签 `support` / `unsupport` 也被 tokenizer 编码；
- 图先经过 `encode_graphs()` 得到图向量；
- 再经过 `projector` 映射到 LLM 词向量空间。

然后看真正的拼接逻辑：

```python
input_ids = texts.input_ids[i][:self.max_txt_len] + eos_user_tokens.input_ids + label_input_ids
inputs_embeds = self.word_embedding(torch.tensor(input_ids).to(self.model.device))

inputs_embeds = torch.cat([bos_embeds, claim_embedding, doc_embedding, inputs_embeds], dim=0)
```

这一行：

```python
torch.cat([bos_embeds, claim_embedding, doc_embedding, inputs_embeds], dim=0)
```

就是图注入 LLM 的核心。

它的含义是：

- 序列最前面先放一个 `BOS` 的 embedding；
- 再放 `claim_kg` 的图向量；
- 再放 `doc_kg` 的图向量；
- 最后才接真正的文本 prompt embedding。

所以图信息是以“额外 embedding token”的形式进入 LLM 的，而不是以普通字符串的形式出现。

### 5.1 把向量拼接代码拆开看

如果你第一次读这一段，可能会觉得这句太浓缩了：

```python
inputs_embeds = torch.cat([bos_embeds, claim_embedding, doc_embedding, inputs_embeds], dim=0)
```

更适合学习的方式，是把它拆成多个中间步骤来看：

```python
# 1) 先准备标签 token：例如 support / unsupport，再在末尾补上 </s>
label_input_ids = (
    labels.input_ids[i][: self.max_new_tokens] + eos_tokens.input_ids
)

# 2) 再准备文本 token：Claim + Document 的文本 token
#    然后拼上用户结束标记 [/INST]
#    最后把正确标签 token 也接到末尾
input_ids = (
    texts.input_ids[i][: self.max_txt_len]
    + eos_user_tokens.input_ids
    + label_input_ids
)

# 3) 把上面的 token id 序列变成普通文本 embedding
text_embeds = self.word_embedding(
    torch.tensor(input_ids).to(self.model.device)
)

# 4) 取出当前样本的 claim 图向量
if claim_embeds.size(0) == batch_size:
    claim_embedding = claim_embeds[i].unsqueeze(0)
else:
    claim_embedding = torch.zeros(self.embed_dim).unsqueeze(0).to(self.model.device)

# 5) 取出当前样本的 doc 图向量
if doc_embeds.size(0) == batch_size:
    doc_embedding = doc_embeds[i].unsqueeze(0)
else:
    doc_embedding = torch.zeros(self.embed_dim).unsqueeze(0).to(self.model.device)

# 6) 把 BOS 的 embedding 放最前面
prefix_embeds = bos_embeds

# 7) 先把 claim 图向量接到 BOS 后面
prefix_plus_claim = torch.cat([prefix_embeds, claim_embedding], dim=0)

# 8) 再把 doc 图向量接到后面
prefix_plus_graphs = torch.cat([prefix_plus_claim, doc_embedding], dim=0)

# 9) 最后再把文本 embedding 接上去
final_inputs_embeds = torch.cat([prefix_plus_graphs, text_embeds], dim=0)
```

虽然源码里没有真的把它写成这么多中间变量，但逻辑上就是这样一步步完成的。

### 5.2 每一步到底在干什么

- 第 1 步：先准备答案部分，也就是标签 token，例如 `support` 或 `unsupport`，并在末尾补上 `</s>`。
- 第 2 步：再准备文本输入部分，即 `Claim + Document` 的 token，然后拼上 `[/INST]`，最后再把标签 token 接到末尾。
- 第 3 步：把第 2 步得到的 token id 序列映射成普通文本 embedding；这时图信息还没有插进去。
- 第 4 步：取第 `i` 条样本对应的 `claim_kg` 图向量，并用 `unsqueeze(0)` 把它变成一行 embedding。
- 第 5 步：取第 `i` 条样本对应的 `doc_kg` 图向量，同样变成一行 embedding。
- 第 6 到 9 步：按顺序把 `BOS -> claim 图向量 -> doc 图向量 -> 文本 embedding` 拼成最终序列。

这里最值得你记住的是：

- 文本部分先被编码成普通 token embedding；
- 两张图不会转成字符串，而是各自变成一个图向量；
- 这两个图向量像“额外 token”一样，被直接插在文本前面。

所以最终输入 LLM 的不是一串纯文本 token，而是一条混合序列：

```text
[BOS embedding(s)]
[claim 图向量]
[doc 图向量]
[文本 prompt 的 embedding]
[[/INST] 的 embedding]
[support / unsupport 的 embedding]
[</s> 的 embedding]
```

如果用一句话概括这段拼接逻辑，那就是：

> 先准备文本答案序列，再把 claim 图向量和 doc 图向量作为两个额外 embedding token 插到文本前面，最后把整条 embedding 序列送进冻结 LLM。

你可以先把一条样本在模型里的输入想象成这样：

```text
[BOS embedding]
[claim 图向量]
[doc 图向量]
[Claim: ... Document: ... 的文本 embedding]
```

这里再给你两个非常重要的阅读结论：

- 当前实现里，每条样本只注入两个图向量，一个代表 `claim_kg`，一个代表 `doc_kg`；
- `src/config.py` 里虽然有 `llm_num_virtual_tokens` 参数，但当前代码路径里并没有真正使用它。

这说明“图虚拟 token 数量可配置”这件事，在当前版本里还没有落到真正的 forward 实现中。

## 6. `forward()` 怎样把它变成一个训练任务

GraphCheck 的训练目标不是传统分类头上的交叉熵，而是“让 LLM 生成正确的标签词”。

先看关键代码：

```python
label_input_ids = labels.input_ids[i][:self.max_new_tokens] + eos_tokens.input_ids
input_ids = texts.input_ids[i][:self.max_txt_len] + eos_user_tokens.input_ids + label_input_ids

label_input_ids = [IGNORE_INDEX] * (inputs_embeds.shape[0]-len(label_input_ids)) + label_input_ids
```

这里可以按时间顺序理解。

### 6.1 文本 prompt 里不止有输入，也有答案位置

`input_ids` 不是只有 `Claim + Document`，它实际上是：

```text
文本 prompt + 用户结束标记 + 标签 token
```

也就是说，训练阶段模型输入序列的尾部，已经把正确答案 token 接上去了。

### 6.2 但 loss 并不是对整个序列都算

`IGNORE_INDEX = -100` 的作用，就是把不需要监督的位置屏蔽掉。

这里的设计含义是：

- 前面的 `BOS`、图向量位置、文本 prompt 位置，都不参与 loss；
- 只有最后标签那一小段 token，会被拿来做监督。

所以这其实是一个很典型的“生成式分类”写法：

- 形式上是生成模型；
- 目标上是在生成 `support` 或 `unsupport`；
- loss 只关心标签 token 是否被生成对了。

最后真正调用 LLM 的部分是：

```python
outputs = self.model(
    inputs_embeds=inputs_embeds,
    attention_mask=attention_mask,
    return_dict=True,
    labels=label_input_ids,
)

return outputs.loss
```

这里直接把 `inputs_embeds` 喂给 LLM，而不是喂 `input_ids`，原因正是前面已经手工把图向量拼进 embedding 序列里了。

## 7. 这套模型真正“在学什么”

如果你现在回头看训练目标，会发现一个很关键的事实：

- LLM 冻结；
- 最终监督是 `support` / `unsupport`；
- 图被压成两个向量再送进 LLM。

所以训练时真正被优化的核心问题是：

> 怎样让 GNN 和 projector 产出的两个图向量，足够帮助冻结的 LLM 在最后生成正确标签？

这可以理解成一种“图条件提示学习”：

- 文本 prompt 提供语言上下文；
- 图向量提供结构化事实补充；
- 冻结 LLM 负责把这些条件综合起来，输出标签。

因此，GraphCheck 当前实现并不是在学一个全新的语言模型，而是在学：

- 哪些图结构信息重要；
- 这些信息应该怎样投影到 LLM 的 embedding 空间；
- 让冻结 LLM 更容易把最终答案生成为 `support` 或 `unsupport`。

## 8. `train.py` 怎样训练它：从数据集到参数更新

理解了模型结构以后，再看 `train.py` 就会清楚很多。

先看模型和优化器这部分：

```python
dataset = KGDataset(args.train_dataset)
idx_split = dataset.get_idx_split()

train_dataset = [dataset[i] for i in idx_split["train"]]
val_dataset = [dataset[i] for i in idx_split["val"]]
test_dataset = [dataset[i] for i in idx_split["test"]]

model = GraphCheck(args=args)

params = [p for _, p in model.named_parameters() if p.requires_grad]
optimizer = torch.optim.AdamW(
    [{"params": params, "lr": args.lr, "weight_decay": args.wd}],
    betas=(0.9, 0.95),
)
```

这段代码最重要的阅读结论是：

- 数据集部分还是沿用上一份笔记里的 `KGDataset` 和 split 文件；
- 模型实例化时已经把 LLM 冻结好了；
- 优化器只拿 `requires_grad=True` 的参数。

因此，虽然 `optimizer` 看起来只是普通 AdamW，但它实际上只会更新：

- `graph_encoder` 的参数；
- `projector` 的参数。

而 LLM 主体不会被更新。

项目还专门打印了可训练参数量：

```python
trainable_params, all_param = model.print_trainable_params()
print(
    f"trainable params: {trainable_params} || all params: {all_param} || trainable%: {100 * trainable_params / all_param}"
)
```

这个打印其实非常值得你自己跑一次，因为它能直观看出“冻结大模型、只训小模块”到底意味着什么。

## 9. 一个 epoch 里到底发生了什么

再看训练循环：

```python
for epoch in range(args.num_epochs):
    model.train()
    epoch_loss = 0.0

    for step, batch in enumerate(train_loader):
        optimizer.zero_grad()
        loss = model(batch)
        loss.backward()
        clip_grad_norm_(optimizer.param_groups[0]["params"], 0.1)
        optimizer.step()
        epoch_loss += loss.item()
```

这一段几乎就是标准训练循环：

1. 从 `train_loader` 拿一个 batch；
2. 调 `model(batch)`，内部进入 `GraphCheck.forward()`；
3. 得到一个标量 loss；
4. 反向传播；
5. 梯度裁剪；
6. `optimizer.step()` 更新参数。

但因为参数冻结策略已经在模型初始化里决定好了，所以这里更新的依然只有图相关模块。

也就是说，在每个 batch 上：

- 文本走进冻结 LLM；
- 图走进可训练 GNN；
- 图表示经过 projector 后插入到 LLM 输入 embedding；
- loss 反向传回 projector 和 GNN；
- 但不会改动大模型主体权重。

## 10. 学习率、验证、早停、checkpoint 是怎样配合的

### 10.1 学习率调整

`train.py` 会在训练过程中调用：

```python
adjust_learning_rate(
    optimizer.param_groups[0],
    args.lr,
    step / len(train_loader) + epoch,
    args,
)
```

对应的实现来自 `src/utils.py`：

```python
def adjust_learning_rate(param_group, LR, epoch, args):
    min_lr = 5e-6
    if epoch < args.warmup_epochs:
        lr = LR * epoch / args.warmup_epochs
    else:
        lr = min_lr + (LR - min_lr) * 0.5 * (
            1.0 + math.cos(math.pi * (epoch - args.warmup_epochs) / (args.num_epochs - args.warmup_epochs))
        )
    param_group["lr"] = lr
```

这说明当前项目的学习率策略是：

- 前期 warmup；
- 后期 cosine 衰减；
- 最低学习率不低于 `5e-6`。

### 10.2 验证和早停

每个 epoch 训练完后，代码会进入验证：

```python
model.eval()
with torch.no_grad():
    for step, batch in enumerate(val_loader):
        loss = model(batch)
        val_loss += loss.item()
```

如果当前 `val_loss` 更好，就保存 best checkpoint：

```python
if val_loss < best_val_loss:
    best_val_loss = val_loss
    _save_checkpoint(model, optimizer, epoch, args, is_best=True)
    best_epoch = epoch
```

如果连续若干个 epoch 都没有提升，就触发早停：

```python
if epoch - best_epoch >= args.patience:
    print(f"Early stop at epoch {epoch}")
    break
```

所以训练的主控逻辑其实很常规：

- 训练集更新参数；
- 验证集决定 best checkpoint；
- `patience` 控制是否提前停止。

### 10.3 checkpoint 保存了什么

这一点在 `src/ckpt.py` 里非常值得注意：

```python
param_grad_dic = {k: v.requires_grad for (k, v) in model.named_parameters()}
state_dict = model.state_dict()
for k in list(state_dict.keys()):
    if k in param_grad_dic.keys() and not param_grad_dic[k]:
        del state_dict[k]
```

这意味着保存 checkpoint 时，项目会主动删掉不需要梯度的参数。

换句话说，best checkpoint 主要保存的是：

- 可训练模块参数；
- 优化器状态；
- 训练配置；
- epoch 编号。

这和整个项目的训练思路是一致的：重点不是保存完整微调后的 LLM，而是保存图相关可训练部分。

## 11. 测试阶段怎么出结果：先生成文本，再从文本里抠标签

训练结束后，`train.py` 会重新加载 best checkpoint，然后在测试集上做生成：

```python
model = _reload_best_model(model, args)
model.eval()

with torch.no_grad():
    output = model.inference(batch)
```

`inference()` 最后会返回生成文本：

```python
outputs = self.model.generate(
    inputs_embeds=inputs_embeds,
    max_new_tokens=self.max_new_tokens,
    attention_mask=attention_mask,
    use_cache=True,
)
pred = self.tokenizer.batch_decode(outputs, skip_special_tokens=True)
```

然后 `src/utils.py` 里的 `get_accuracy(...)` 会用正则从预测文本里找：

```python
matches = re.findall(r"unsupport|support", str(pred).strip(), re.IGNORECASE)
```

也就是说，最终评估不是直接比较分类 logits，而是：

1. 先让模型生成文本；
2. 再从文本里抽取 `support` 或 `unsupport`；
3. 最后计算 balanced accuracy。

这再次说明：GraphCheck 的训练和评估范式，都更接近“生成式分类”而不是“传统分类头”。

## 12. 当前实现里几个很值得你注意的细节

这一节不是说代码一定有问题，而是帮助你带着“源码阅读者”的视角去看训练方法。

### 12.1 `grad_steps` 这个参数目前不是标准梯度累积

`src/config.py` 里定义了：

```python
parser.add_argument("--grad_steps", type=int, default=2)
```

但在 `train.py` 里，代码每个 batch 都会执行：

```python
optimizer.zero_grad()
loss.backward()
optimizer.step()
```

而 `grad_steps` 只在下面这段里参与：

```python
if (step + 1) % args.grad_steps == 0:
    adjust_learning_rate(...)
```

这意味着从当前实现看：

- 参数仍然是每个 batch 更新一次；
- `grad_steps` 并没有实现标准意义上的“累积若干步再更新”；
- 它更像是在影响学习率调整节奏和部分统计逻辑。

所以你读训练代码时，不要仅凭参数名就默认它做了梯度累积，最好看实际调用位置。

### 12.2 `llm_num_virtual_tokens` 目前没有真正进入 forward 路径

`src/config.py` 里有：

```python
parser.add_argument("--llm_num_virtual_tokens", type=int, default=4)
```

但当前代码里，真正被拼进输入序列的图向量只有两个：

- 一个 `claim_embedding`；
- 一个 `doc_embedding`。

所以如果你以后想扩展这个项目，一个很自然的问题就是：

> 是否要把一张图映射成多个虚拟 token，而不是只压成一个向量？

但这是后续可能的设计方向，不是当前代码已经实现的行为。

## 13. 学完这一阶段后，你脑中应该留下什么

如果你已经读到这里，建议你最后检查自己能不能复述下面这条主线：

- `claim_kg` 和 `doc_kg` 先进入 GNN；
- GNN 先产出节点表示，再做平均池化得到图表示；
- 图表示经过 projector 映射到 LLM embedding 空间；
- 这两个图向量被直接拼到文本 embedding 前面；
- 冻结 LLM 在这些条件上生成 `support` / `unsupport`；
- 训练时真正更新的是 GNN 和 projector，而不是 LLM 主体。

如果你能把这条链自己讲出来，说明你已经抓住 GraphCheck 当前实现的模型本质了。
