# GraphCheck 数据预处理详解

> 这份文档的目标不是只告诉你“代码做了什么”，而是尽量把 **数据预处理每一步为什么这样做、输入输出分别是什么、这些产物后面怎样被训练与测试使用** 讲清楚。  
> 建议你一边阅读，一边对照这些源码文件：`graph_building.sh`、`graph_build.py`、`dataset/utils/modeling.py`、`dataset/utils/dataset.py`、`dataset/utils/collate.py`、`train.py`、`inference.py`。

## 1. 先给结论：这个项目的数据预处理到底在做什么

如果只用一句话概括，GraphCheck 的数据预处理就是：

- 把原始样本中的 `doc_kg` / `claim_kg` 三元组；
- 转成标准化的节点表和边表；
- 再把节点文本、边文本编码成向量；
- 最终保存成 PyTorch Geometric 可以直接读取的图对象；
- 同时生成训练 / 验证 / 测试索引；
- 供后面的训练与测试直接读取，而不在运行时重复做这些昂贵操作。

也就是说，这个项目的预处理不是简单的“清洗文本”，而是在做一件更重要的事：

**把“文本形式的知识图谱”变成“神经网络可计算的图表示”。**

---

## 2. 先建立全局数据流

你可以先把 GraphCheck 的数据链路记成下面这条主线：

```text
原始 .pkl
  -> 读取 doc_text / claim_text / doc_kg / claim_kg / label
  -> graph_build.py
  -> nodes/*.csv + edges/*.csv
  -> 文本编码
  -> graphs/*.pt
  -> split/*.txt
  -> KGDataset
  -> collate_fn
  -> train.py / inference.py
  -> GraphCheck
```

再压缩成更短的一句话就是：

```text
原始样本 -> 图预处理产物 -> DataLoader -> 模型
```

理解这条主线之后，你再看每个函数，就不会迷失在局部代码里。

---

## 3. 预处理涉及哪些核心文件

这一部分建议你先建立“文件职责地图”。

### 3.1 `graph_building.sh`

它是一个批处理入口脚本，负责按数据集逐个调用 `graph_build.py`。

它做的事情很简单：

- 列出要处理的数据集名字；
- 对每个数据集执行一次 `python graph_build.py --data_name <dataset>`。

所以它的作用不是“真正处理图”，而是**统一调度**。

### 3.2 `graph_build.py`

这是整个预处理过程的核心文件。

它主要做三件事：

1. 把 KG 三元组转成节点表和边表；
2. 把节点 / 边文本编码成向量图；
3. 生成 train / val / test 划分文件。

### 3.3 `dataset/utils/modeling.py`

它负责“文本转向量”，也就是：

- 加载预训练句向量模型；
- 调 tokenizer；
- 调 transformer 编码；
- 做 mean pooling；
- 输出节点和边的 embedding。

### 3.4 `dataset/utils/dataset.py`

它不负责“生成图”，而是负责在训练和测试时：

- 重新读取 `.pkl` 中的文本和标签；
- 读取预处理好的 `.pt` 图对象；
- 把它们重新组合成一条可训练样本。

### 3.5 `dataset/utils/collate.py`

它负责把多条样本拼成一个 batch，尤其是：

- 文本字段按普通列表拼；
- 图字段用 PyG 的 `Batch.from_data_list(...)` 合并。

### 3.6 `train.py` / `inference.py`

这两个文件都不做图构建；它们只消费预处理产物。

- `train.py` 读取 split，构造 train / val / test；
- `inference.py` 直接读取整个评测集对应的图和文本，跑推理。

---

## 4. 原始输入数据长什么样

GraphCheck 的原始样本不是“只有文本”的表，也不是“已经编码好”的图，而是一种混合的中间形式。

每条样本至少包含这些字段：

- `doc_text`：原始文档文本
- `claim_text`：待核查声明
- `doc_kg`：从文档抽取出的 KG 三元组列表
- `claim_kg`：从声明抽取出的 KG 三元组列表
- `label`：监督标签，通常是 `0/1`

一个非常典型的样本可以抽象成：

```python
{
    "doc_text": "Prime Minister Rishi Sunak's Conservative government ...",
    "claim_text": "The government aims to reform welfare.",
    "doc_kg": [
        ("Rishi Sunak", "leads", "Conservative government"),
        ("Conservative government", "struggles to bolster", "economy"),
    ],
    "claim_kg": [
        ("government", "aims to", "reform welfare")
    ],
    "label": 0,
}
```

这里要抓住一个关键事实：

- `doc_text` / `claim_text` 仍然会在后续训练中被使用；
- `doc_kg` / `claim_kg` 也会被使用；
- 也就是说，GraphCheck 不是“只用图”的系统，而是“文本 + 图”同时使用。

这直接决定了后续预处理的目标不是替代文本，而是把 KG 补充成额外的结构化输入。

---

## 5. `extracted_KG` 目录到底是什么

`dataset/extracted_KG/<dataset>/` 可以理解成 **预处理结果仓库**。

它不是模型参数目录，也不是最终评测结果目录，而是“图预处理阶段产物”的集中存放位置。

典型结构如下：

```text
dataset/extracted_KG/<dataset>/
  <dataset>.pkl
  nodes/
    claim/
      0.csv
      1.csv
    doc/
      0.csv
      1.csv
  edges/
    claim/
      0.csv
      1.csv
    doc/
      0.csv
      1.csv
  graphs/
    claim/
      0.pt
      1.pt
    doc/
      0.pt
      1.pt
  split/
    train_indices.txt
    val_indices.txt
    test_indices.txt
```

可以把这几类文件分别理解成：

- `<dataset>.pkl`：原始样本元数据
- `nodes/*.csv`：节点表
- `edges/*.csv`：边表
- `graphs/*.pt`：真正喂给 GNN 的图对象
- `split/*.txt`：训练 / 验证 / 测试划分索引

换句话说：

- 原始 `.pkl` 负责保存“语义内容”；
- `nodes/edges` 负责保存“中间结构”；
- `graphs` 负责保存“可计算图”；
- `split` 负责保存“数据切分”。

---

## 6. 预处理入口：`graph_building.sh` 如何调度

这个脚本本身很短，但它体现了 GraphCheck 的数据预处理组织方式。

它的基本逻辑是：

```bash
for dataset in datasets:
    python graph_build.py --data_name "$dataset"
```

这说明两件事：

1. 每个数据集的图构建是独立的；
2. `graph_build.py` 被设计成“处理单个数据集”的通用入口。

这类设计的优点是：

- 便于单独重建某一个数据集；
- 某个数据集处理失败时，不影响代码结构本身；
- 也方便后续扩展新的评测集。

但这里有一个实际使用时很值得注意的点：

- 当前 `graph_building.sh` 里列的是多个评测集；
- `train.sh` 默认训练集却是 `MiniCheck_Train`；
- 这意味着如果你要完整训练，通常还需要显式跑一次：

```bash
python graph_build.py --data_name MiniCheck_Train
```

也就是说，训练集和评测集的图预处理入口思路是一致的，只是脚本里列的默认集合不完全相同。

---

## 7. `graph_build.py` 的执行流程总览

如果你只看函数名，可以把 `graph_build.py` 的主流程概括成：

```python
dataset = pd.read_pickle(...)
doc_kgs = dataset["doc_kg"]
claim_kgs = dataset["claim_kg"]

step_one()   # 三元组 -> nodes/edges csv
step_two()   # nodes/edges text -> .pt graph
generate_split(...)  # 写出 train/val/test 索引
```

所以它不是一个复杂的隐式框架，而是一个非常明确的三阶段流水线：

1. 结构化
2. 向量化
3. 划分化

下面逐步展开。

---

## 8. 第一阶段：读取原始 `.pkl`

在 `graph_build.py` 的主程序里，会先读取：

```python
dataset = pd.read_pickle(f"{path}/{data_name}/{data_name}.pkl")
```

然后取出：

```python
doc_kgs = dataset["doc_kg"]
claim_kgs = dataset["claim_kg"]
labels = dataset["label"]
```

这一阶段的本质不是变换，而是 **建立后续流水线的输入集合**。

为什么只先取 `doc_kg` / `claim_kg`？

因为 `graph_build.py` 的职责是“图预处理”，此时它重点关心的是 KG 本身，而不是文档和声明文本如何送进 LLM。

但是 `label` 仍然会被保留在原始 `.pkl` 中，后面训练时还要读取它。

所以这里要区分：

- `graph_build.py` 聚焦的是图部分；
- 整个系统最终使用的仍是文本 + 图 + 标签三者组合。

---

## 9. 第二阶段：`textualize_graph()` 如何把三元组变成图结构

这是整个预处理中最基础、最关键的一步。

### 9.1 输入是什么

函数输入 `graph` 是一个三元组列表，每个元素一般形如：

```python
(head, relation, tail)
```

例如：

```python
[
    ("Rishi Sunak", "leads", "Conservative government"),
    ("Conservative government", "struggles to bolster", "economy"),
]
```

### 9.2 输出是什么

输出是两张表：

- `nodes`：节点表
- `edges`：边表

其中：

- 节点表记录“有哪些实体，以及它们的编号”；
- 边表记录“哪些实体之间有关系，以及关系文本是什么”。

### 9.3 具体执行逻辑

`textualize_graph()` 做了这几件事：

1. 如果图为空，直接返回空的节点表和边表；
2. 用字典 `nodes` 给实体分配唯一 `node_id`；
3. 遍历每个三元组 `(src, edge_attr, dst)`；
4. 把头实体、关系、尾实体都转成小写并去掉首尾空格；
5. 为没见过的实体分配编号；
6. 把关系记录成一条边；
7. 最后统一转成 DataFrame。

### 9.4 为什么要转小写和 strip

这是标准化的一部分。

如果不做这一步，下面这些文本可能被错误地当成不同节点：

- `Rishi Sunak`
- `rishi sunak`
- `Rishi Sunak `

而代码通过：

- `lower()`
- `strip()`

尽量把这些形式归一到同一个节点。

这样做的原理是：

- 实体节点在图里应该尽量代表“语义单位”，而不是字符串表面差异；
- 如果同一个实体因为大小写或空格不同被拆成多个节点，图结构会被污染。

### 9.5 为什么节点要去重，边却不去重

节点用字典去重，是因为一个实体应该只占一个节点编号。

但边这里没有额外做去重，这意味着：

- 如果两个三元组完全重复，边表里可能出现重复边；
- 如果同一对节点之间有多种关系，也会保留多条边。

这其实是合理的，因为边本来就可能是多重关系，而不是简单邻接。

### 9.6 空图怎么处理

如果 `graph` 为空，函数会返回空表。

这一步非常重要，因为真实数据中可能存在：

- 某条 claim 没抽出任何三元组；
- 某条 document 的 KG 抽取为空。

如果没有空图兜底，后面保存 CSV、编码 embedding、构造 `.pt` 时都可能报错。

所以这里的空图分支本质上是在保证预处理流水线的健壮性。

---

## 10. 第三阶段：`step_one()` 如何把每条样本落盘为 `nodes/edges`

`step_one()` 的作用可以概括成：

**把每条样本里的 claim 图和 doc 图，拆成可以独立保存、独立检查、独立编码的结构化文件。**

### 10.1 它做了哪些目录准备

它会先创建：

- `nodes/claim/`
- `nodes/doc/`
- `edges/claim/`
- `edges/doc/`

这是在告诉你：

- claim 图和 doc 图是分开存的；
- 节点表和边表也是分开存的；
- 一个样本实际上会对应四个 CSV 文件。

### 10.2 它如何遍历样本

核心循环是：

```python
for i, (claim_kg, doc_kg) in enumerate(zip(claim_kgs, doc_kgs)):
```

这说明样本编号 `i` 是整个项目里非常重要的“主键”。

后面这些文件都会共享同一个编号体系：

- 原始 `.pkl` 第 `i` 行
- `nodes/.../{i}.csv`
- `edges/.../{i}.csv`
- `graphs/.../{i}.pt`
- split 文件中的索引 `i`

也就是说，**样本编号就是所有模态之间的对齐桥梁。**

### 10.3 写出的 CSV 长什么样

假设某个 claim 图只有一条三元组：

```python
[("government", "aims to", "reform welfare")]
```

那么它可能产生：

`nodes/claim/0.csv`

```text
node_id,node_attr
0,government
1,reform welfare
```

`edges/claim/0.csv`

```text
src,edge_attr,dst
0,aims to,1
```

### 10.4 为什么不直接一步生成 `.pt`

这是很多人第一次看代码时会疑惑的地方。

原因主要有四个：

1. **便于调试**：CSV 是人能直接看的，`.pt` 不是。
2. **便于复用**：如果图文本结构没变，只想改 embedding 模型，可以直接从 CSV 重新编码。
3. **解耦步骤**：把“图结构规范化”和“文本向量化”拆开，逻辑更清楚。
4. **便于定位错误**：如果出问题，可以判断是三元组解析错了，还是 embedding 编码错了。

所以 `step_one()` 的本质是先构造一个“可解释、可检查”的中间层。

---

## 11. 第四阶段：`step_two()` 如何把文本图编码成向量图

这是最核心的计算阶段。

如果 `step_one()` 把图从“原始三元组”变成了“结构化文本图”，那么 `step_two()` 做的就是：

**把结构化文本图变成真正可以输入 GNN 的数值图。**

### 11.1 它的总体流程

你可以把 `step_two()` 理解成下面这条链：

```text
nodes.csv / edges.csv
  -> 读取节点文本和边文本
  -> tokenizer
  -> 预训练句向量模型
  -> 得到节点向量 x 和边向量 e
  -> 用 src/dst 构造 edge_index
  -> 组装 PyG Data
  -> 保存为 .pt
```

### 11.2 `load_model()` 在做什么

`step_two()` 开始时会调用：

```python
model, tokenizer, device = load_model()
```

这一步来自 `dataset/utils/modeling.py`，它会：

- 加载预训练句向量模型 `sentence-transformers/all-roberta-large-v1`
- 加载对应 tokenizer
- 把模型放到 GPU 或 CPU
- 设为 `eval()` 模式

注意这里加载的不是 GraphCheck 主模型，而是一个**专门用于把文本编码成 embedding 的句向量模型**。

### 11.3 `text2embedding(...)` 输入输出分别是什么

当代码执行：

```python
x = text2embedding(model, tokenizer, device, nodes.node_attr.tolist())
e = text2embedding(model, tokenizer, device, edges.edge_attr.tolist())
```

它做的事情分别是：

- 把所有节点文本编码成节点特征矩阵 `x`
- 把所有边关系文本编码成边特征矩阵 `e`

如果当前图中有：

- 3 个节点
- 2 条边

那么最后通常会得到：

- `x.shape == [3, 1024]`
- `e.shape == [2, 1024]`

这里的 1024 来自所用句向量模型的隐藏维度。

### 11.4 文本编码的原理是什么

`load_text2embedding()` 的内部逻辑可以按下面顺序理解：

1. 输入是字符串列表；
2. tokenizer 把每条字符串变成 token id；
3. DataLoader 分批喂给 transformer；
4. transformer 输出每个 token 的上下文向量；
5. `mean_pooling()` 对同一条文本的 token 向量做平均；
6. `F.normalize(...)` 做 L2 归一化；
7. 返回一个 `[文本条数, 1024]` 的矩阵。

换句话说，节点和边的特征不是人工设计的 one-hot，也不是随机初始化，而是：

**用预训练语言模型得到的语义向量。**

这样做的好处是：

- `government` 和 `administration` 这类语义相近词，向量可能也更接近；
- `leads` 和 `heads` 这类关系词，模型能带入一定语义先验；
- GNN 后续传播时，不是只看结构，还能看每个节点 / 边“代表什么”。

### 11.5 为什么节点和边都要编码

因为在这个项目里：

- 节点不是匿名 ID，而是实体文本；
- 边也不是无类型连接，而是有关系文本的有语义边。

如果你只编码节点、不编码边，那么：

- 图知道谁和谁相连；
- 但不知道“以什么关系相连”。

而编码边关系之后，模型就能区分：

- `government -> supports -> policy`
- `government -> rejects -> policy`

虽然两张图结构形式可能一样，但边语义完全不同。

### 11.6 `edge_index` 的作用是什么

这一句：

```python
edge_index = torch.LongTensor([edges.src, edges.dst])
```

构造的是 PyTorch Geometric 中常见的图连接格式。

它本质上是一个 `[2, num_edges]` 的整数张量：

- 第 0 行：每条边的起点编号
- 第 1 行：每条边的终点编号

例如：

```text
src: 0, 1
dst: 1, 2
```

会变成：

```python
edge_index = [[0, 1],
              [1, 2]]
```

GNN 后续就是靠它知道消息该沿哪些边传播。

### 11.7 最后保存的 `.pt` 图对象是什么

代码会组装：

```python
Data(x=x, edge_index=edge_index, edge_attr=e, num_nodes=len(nodes))
```

也就是说，一张图至少包含：

- `x`：节点特征矩阵
- `edge_index`：图结构连接关系
- `edge_attr`：边特征矩阵
- `num_nodes`：节点数

这已经是 PyG 可以直接输入 GNN 的标准图对象格式。

### 11.8 空节点或空边怎么处理

`load_text2embedding()` 里有一个重要兜底：

```python
if len(text) == 0:
    return torch.zeros((0, 1024))
```

它的作用是：

- 如果当前图没有节点文本或边文本；
- 不去强行调用 tokenizer 和 transformer；
- 直接返回一个空的 embedding 矩阵。

这说明作者考虑到了预处理中的异常样本和空图场景。

## 12. 第五阶段：`generate_split()` 如何生成数据划分

图文件生成好之后，预处理还没有结束，因为训练还需要知道：

- 哪些样本属于训练集；
- 哪些属于验证集；
- 哪些属于测试集。

`generate_split()` 做的就是这件事。

### 12.1 它是如何切分的

代码逻辑是：

1. 先生成从 `0` 到 `N-1` 的全部样本索引；
2. 第一次切分拿出 `40%` 作为临时集合；
3. 剩余 `60%` 作为训练集；
4. 再把那 `40%` 平分成验证集和测试集。

所以最终比例是：

- train = 60%
- val = 20%
- test = 20%

### 12.2 为什么切的是“索引”而不是“数据”

因为样本本体已经分散保存在多个位置：

- 文本和标签在 `.pkl`
- 图对象在 `graphs/claim/*.pt` 与 `graphs/doc/*.pt`

这些文件都是靠样本编号对齐的，所以只要保存：

- 哪些编号属于 train
- 哪些编号属于 val
- 哪些编号属于 test

就足够了。

这种方式的优点是：

- 节省空间；
- 复现方便；
- 不需要复制数据；
- 文本和图天然保持同步。

### 12.3 `random_state=42` 的意义

这代表划分是可复现的。

也就是说，同一份数据重复执行同样的切分逻辑，会得到相同划分。

这对科研实验尤其重要，因为：

- 不同划分可能影响结果；
- 固定划分有助于公平比较模型。

### 12.4 这里没有分层采样

代码里没有 `stratify=labels`。

这意味着它做的是普通随机划分，而不是按标签比例严格保持分布一致的分层采样。

这在数据不平衡场景下可能带来一定风险：

- train / val / test 的正负样本比例可能有偏差；
- 评测波动可能更大。

不过这不是“代码错误”，而是当前实现的设计选择。

---

## 13. 预处理结束后，训练阶段到底读取了什么

很多人以为“图预处理结束后，训练时只读图”，其实不是。

训练阶段读的是一条 **重新组装后的多模态样本**。

### 13.1 `KGDataset` 先读什么

`dataset/utils/dataset.py` 里的 `get_dataset(dataset_name)` 会先读取：

```python
result_df = pd.read_pickle(...)
docs = result_df["doc_text"].values
claims = result_df["claim_text"].values
doc_kgs = result_df["doc_kg"].values
claim_kgs = result_df["claim_kg"].values
labels = result_df["label"].values
```

也就是说，训练时原始 `.pkl` 依然有价值，不会被 `.pt` 图完全替代。

### 13.2 `__getitem__()` 如何组装一条样本

对某个索引 `i`，它会同时取出：

- 文档文本 `doc`
- claim 文本 `claim`
- claim 图对象 `graphs/claim/{i}.pt`
- doc 图对象 `graphs/doc/{i}.pt`
- 标签 `label`

然后拼成：

```python
{
    "text": prompt + claim + document,
    "claim_kg": claim_kg,
    "doc_kg": doc_kg,
    "label": "support" or "unsupport",
    ...
}
```

所以 GraphCheck 的训练样本并不是“图分类样本”，而是：

**带文本输入、带两张图输入、带自然语言标签的多模态样本包。**

### 13.3 为什么保留 `claim_kg_text` / `doc_kg_text`

虽然当前主模型核心使用的是 `.pt` 图对象，但代码依然保留三元组文本字段。

这有几个潜在价值：

- 便于调试时核对图和原始 KG 是否一致；
- 便于后续扩展需要直接读取 KG 文本的功能；
- 便于分析某条样本时同时观察结构表示和原始三元组表示。

---

## 14. `collate_fn` 如何把多条样本拼成 batch

DataLoader 默认比较擅长拼普通张量，但这里每条样本里有：

- 字符串
- 标签
- PyG 图对象

所以项目定义了自定义 `collate_fn`。

它的逻辑可以拆成两层：

### 14.1 第一层：把所有字段先收集成列表

例如一个 batch 中有 4 条样本，那么：

- `text` 变成长度为 4 的列表
- `label` 变成长度为 4 的列表
- `claim_kg` 变成 4 张图组成的列表
- `doc_kg` 变成 4 张图组成的列表

### 14.2 第二层：把图列表变成 PyG 的批量图

```python
batch['claim_kg'] = Batch.from_data_list(batch['claim_kg'])
batch['doc_kg'] = Batch.from_data_list(batch['doc_kg'])
```

这一步的意义是：

- 将多张离散图拼成一个 batched graph；
- 附带生成 `batch` 向量，标记每个节点属于哪一张原始图；
- 使后续 GNN 可以一次处理一个 mini-batch。

这也解释了为什么 `model/graphcheck.py` 里后续能用：

```python
scatter(node_embeds, claim_kg.batch, dim=0, reduce='mean')
```

因为 `claim_kg.batch` 正是 PyG 在批处理阶段自动提供的“节点归属图编号”。

## 15. 训练时预处理产物怎样接到模型前向中

现在把视角从预处理切到模型使用阶段。

### 15.1 `train.py` 怎样消费 split

训练时会执行：

```python
dataset = KGDataset(args.train_dataset)
idx_split = dataset.get_idx_split()

train_dataset = [dataset[i] for i in idx_split["train"]]
val_dataset = [dataset[i] for i in idx_split["val"]]
test_dataset = [dataset[i] for i in idx_split["test"]]
```

这说明：

- 训练集、验证集、测试集不是独立文件；
- 它们是“同一套样本 + 不同索引子集”；
- split 文件真正决定了训练时读哪些编号。

### 15.2 `GraphCheck.encode_graphs()` 如何消费 `.pt` 图对象

训练时，batch 里的 `claim_kg` 和 `doc_kg` 会进入 GNN：

```python
claim_n_embeds, _ = self.graph_encoder(
    claim_kg.x, claim_kg.edge_index.long(), claim_kg.edge_attr
)
```

这里你可以直接对应回预处理阶段的产物：

- `claim_kg.x` 来自节点文本 embedding
- `claim_kg.edge_index` 来自边表中的 src/dst
- `claim_kg.edge_attr` 来自关系文本 embedding

所以预处理阶段生成的 `.pt` 图对象，正好就是 GNN 的直接输入。

### 15.3 为什么最后还要做图级池化

GNN 编码后先得到的是节点级表示，但 GraphCheck 不是把每个节点都单独送给 LLM，而是：

- 对每张 claim 图池化成一个图向量
- 对每张 doc 图池化成一个图向量

然后再投影到 LLM embedding 空间，插入到文本输入序列里。

所以预处理阶段的图对象虽然是节点级图，但最终在 GraphCheck 里会变成两个图级语义摘要。

## 16. 测试 / 推理时预处理产物怎样被使用

测试时流程和训练很像，区别主要不在数据表示，而在是否使用标签监督和是否更新参数。

`inference.py` 中会：

```python
dataset = KGDataset(args.dataset_name)
test_loader = DataLoader(dataset, ..., collate_fn=collate_fn)
```

这说明推理阶段同样会读取：

- 文本
- `claim_kg.pt`
- `doc_kg.pt`

也就是说：

**测试数据也包括图。**

预处理阶段生成的图文件不是训练专用，而是训练和评测共用的基础设施。

---

## 17. 为什么要把预处理放在训练前离线完成

这是理解整个设计原理时非常关键的一点。

如果不预处理，理论上也可以在训练时现做这些步骤：

- 读三元组
- 生成节点表 / 边表
- 编码节点 / 边文本
- 构图

但这样会带来严重问题：

### 17.1 文本编码成本很高

节点和边的文本 embedding 是通过大型 transformer 做的。

如果每个 epoch、每个 batch 都重新编码：

- 速度会非常慢；
- GPU 显存和算力消耗会很高；
- 训练成本会显著上升。

### 17.2 图结构不会在每个 epoch 改变

对于固定数据集来说：

- 第 7 条样本的 `claim_kg` 是固定的；
- 对应的节点、边、图结构也是固定的。

既然输入图不变，就没必要每轮重算一次。

### 17.3 可复现性更好

把图对象提前保存后：

- 后续训练直接读同一份 `.pt`；
- 多次实验用的是同一套图输入；
- 结果更稳定，也更方便排查问题。

### 17.4 工程上更易调试

一旦预处理和训练解耦：

- 预处理报错，只查 `graph_build.py`；
- 训练报错，只查训练图读取或模型前向；
- 问题边界更清晰。

所以从原理上说，预处理阶段承担的是：

**把昂贵、稳定、不需要每轮重算的工作提前离线化。**

---

## 18. 为什么 GraphCheck 要同时保留文本和图

这属于“预处理设计背后的建模动机”。

仅使用长文档文本时，LLM 在事实核查里可能遇到的问题包括：

- 长文本中事实线索分散；
- 实体和关系跨句出现，不容易显式对齐；
- 某些支持 / 矛盾关系需要结构化视角才能更稳定地捕捉。

而 KG 预处理做的事情是：

- 把分散在文本里的实体和关系抽取出来；
- 用图把它们连起来；
- 再用 GNN 在图上做关系传播；
- 最后把图语义作为额外信息注入 LLM。

所以预处理并不是“为了把文本替换掉”，而是：

**把文本中的关系结构显式化。**

这也解释了为什么预处理阶段要特别认真地构建：

- 节点文本
- 边文本
- 图连接关系

因为它们共同决定了后续图表示质量。

## 19. 这个预处理流程有哪些关键假设

读代码时你会发现，这套流程隐含了几条默认假设。

### 19.1 KG 三元组已经存在

仓库里的 `graph_build.py` 并不负责从自然语言中抽取三元组，它假设：

- `.pkl` 里已经有 `doc_kg`
- `.pkl` 里已经有 `claim_kg`

所以“抽取 KG”是更早一步的数据构造工作，而不是当前仓库里这个预处理脚本的职责。

### 19.2 节点和边都可以用文本 embedding 表示

也就是说，当前实现默认：

- 实体语义可以由实体名向量近似；
- 关系语义可以由关系短语向量近似。

这是一种文本图建模思路，而不是符号图谱嵌入思路。

### 19.3 样本索引是一切对齐的基础

整个工程默认：

- `.pkl` 第 `i` 行
- `graphs/claim/{i}.pt`
- `graphs/doc/{i}.pt`
- split 中的编号 `i`

必须严格对应。

一旦这个对应关系出错，整个训练和评测都会错位。

## 20. 这个预处理流程有哪些值得注意的实现细节

### 20.1 路径组织上有一个实际注意点

当前训练 / 读取代码默认从：

```text
dataset/extracted_KG/<dataset>/<dataset>.pkl
```

读取 `.pkl`。

但仓库中也能看到：

```text
dataset/<dataset>/<dataset>.pkl
```

这说明你在真正运行时，需要确认：

- 预处理脚本期望的输入路径是否已经准备好；
- 原始 `.pkl` 是否放在代码实际读取的位置。

否则会在 `pd.read_pickle(...)` 这一步直接失败。

### 20.2 `labels` 在 `graph_build.py` 中被读出但未直接参与图构建

这是因为标签不影响图结构本身。

图预处理阶段的目标是：

- 构建输入表示；
- 不负责构建监督信号。

标签的真正使用发生在后面的 `KGDataset` 和模型训练里。

### 20.3 句向量模型与主模型解耦

节点 / 边 embedding 使用的是 `sentence-transformers/all-roberta-large-v1`，而不是训练时的大语言模型。

这样做的好处是：

- 预处理更加独立；
- 不依赖超大 LLM 做图输入构造；
- 图表示可以在训练前固定下来。

### 20.4 图构建是逐样本进行的

这不是把整个数据集拼成一张大图，而是：

- 每条样本一张 `claim` 图
- 每条样本一张 `doc` 图

这符合任务本身的结构，因为每条事实核查样本本来就对应“一条声明 + 一篇文档”。

## 22. 最后把整条因果链再复述一遍

如果你已经读完这份文档，建议你检查自己能不能完整复述下面这条链：

1. 原始 `.pkl` 中既有文本，也有 KG 三元组和标签；
2. `graph_build.py` 先把三元组变成节点表和边表；
3. 再把节点文本和关系文本编码成向量；
4. 用 `edge_index + x + edge_attr` 组装成 PyG 图对象；
5. 同时把样本编号切分成 train / val / test 三份；
6. `KGDataset` 在训练 / 测试时重新读回文本、图和标签；
7. `collate_fn` 把多条样本变成一个包含文本和图的 batch；
8. 模型中的 GNN 直接消费预处理好的 `.pt` 图；
9. claim 图和 doc 图最后各自被压成一个图向量，再与文本一起送入 LLM。

如果这 9 步你已经能顺下来，说明你已经不只是“知道有哪些文件”，而是真的理解了 GraphCheck 的数据预处理主线。

## 23. 一句话总结这套预处理的核心原理

GraphCheck 的数据预处理，本质上是在做下面这件事：

**把原始样本中的 KG 三元组，转成兼具结构信息与语义信息的图表示，并把这些稳定、昂贵的计算提前离线完成，以便训练和测试阶段直接高效复用。**

这句话如果你已经真正理解，后面再看 `model/gnn.py` 和 `model/graphcheck.py` 时，就会非常顺。
